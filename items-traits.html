// CURRENT MODEL · INVENTORY

Items.
Traits + Archetypes.

Item Data Asset теперь собирается из archetype chain и narrow traits. В editor правятся authoring traits, а runtime читает уже построенные fragments/actions.

01

// Зачем это

Старый Item Data Asset показывал дизайнеру все возможные свойства предмета сразу — включая те, что никогда не будут заполнены. Новая модель делит предмет на archetype defaults, inherited trait overrides и local capability traits.

Traits — это editor-only authoring layer. Каждый effective trait строит data-only fragments и, если нужно, runtime actions. Итог сохраняется на asset как MergedFragments и MergedRuntimeActions, поэтому runtime не зависит от editor-only authoring data.

01

Меньше шума

Дизайнер видит только те поля, которые относятся к этому предмету. Не нужно угадывать, что заполнять.

02

Наследование

Archetype задаёт общий состав предмета. Конкретный DA переопределяет только нужные inherited traits.

03

Готовность к росту

Новая способность (weapon, scannable, capability X) — новый trait. Добавляем туда, где нужно. Не трогаем то, где не нужно.

04

Runtime чище

Игровой код читает построенные fragments через lookup API, а не authoring traits из editor.

Три слоя

Archetype задаёт общий template. Archetype Traits — local overrides inherited trait slots, с reset-to-default к parent values. Traits — локальные additions, например stackable/container/scannable capability. Build step складывает effective traits в fragments/actions.

Что произошло

Effective trait set строится parent-first. Local trait exact class заменяет inherited trait только в Archetype Traits; новые capabilities добавляются в Traits. Потом build step создаёт один merged fragment list для runtime.

03

// Куда это растёт

Новая capability добавляется narrow trait'ом. Если capability нужна всем предметам archetype family — она живёт в archetype. Если нужна только одному предмету — она лежит в local Traits.

04

// Что с существующими предметами

TL;DR для дизайнеров

Ничего делать не надо. Открыл предмет в editor — он уже в новом виде. Сохранил — формат закрепился.

Старые Item Data Asset мигрируются автоматически на загрузке: legacy root fields превращаются в archetype overrides или local capability traits, затем пересобираются generated fragments/actions. Старые поля пока остаются в классе только для десериализации старых asset-файлов.

05

// FAQ

А если нужно для одного предмета своё поведение, которого нет в archetype?
Если это variation inherited trait'а — правь его в Archetype Traits. Если это новая orthogonal capability — добавь narrow trait в Traits. Runtime side effects должны идти через trait runtime actions/components; fragments остаются data-only.
Сломаются ли сейвы?
Нет. Сейв ссылается на DA по пути, миграция идёт прозрачно.
Почему два списка: Archetype Traits и Traits?
Archetype Traits — это local copies inherited slots, у них есть parent default для reset. Traits — это local-only additions. Trait того же exact class, что уже есть в archetype, должен быть override, а не addition.
Почему не сделать просто разные подклассы Item Data Asset?
Способности ортогональны: один предмет может быть одновременно обычным, stackable, scannable и special-purpose item'ом. Подклассы — это однонаследная иерархия, которая не выражает «и то, и это, и третье». Traits/archetypes дают композицию и inherited defaults без разрастания root DA fields.
Куда писать с вопросами / правками?
Через внутренний канал команды.
