TECH NOTE ARCH · ITEMS COMPOSITION ITM-001 · REV 03
// SYSTEM UPGRADE · INVENTORY

Items.
Traits вместо россыпи полей.

Item Data Asset перестаёт быть «свалкой свойств». Свойства переезжают в traits — каждый отвечает за свой аспект предмета. Editor становится в разы чище.

01

// Зачем это

Сейчас Item Data Asset показывает дизайнеру все возможные свойства предмета сразу — включая те, что никогда не будут заполнены. У обычного предмета есть поля, относящиеся к контейнерам. У валюты есть поля экипировки. И так дальше.

Когда подъедут новые capability (weapon, scannable, специальные категории, quest-привязки) — добавится ещё столько же полей. И большинство из них так же будут пустыми у большинства предметов. Текущая модель упирается в этот рост: добавить, например, винтовку как DA сейчас просто некуда — FamilyId, Recoil, Stance не имеют места в существующих полях.

01

Меньше шума

Дизайнер видит только те поля, которые относятся к этому предмету. Не нужно угадывать, что заполнять.

02

Меньше ошибок

Невозможно «забыть» заполнить — нужные поля приходят набором, отсутствующих просто нет.

03

Готовность к росту

Новая способность (weapon, scannable, capability X) — новый trait. Добавляем туда, где нужно. Не трогаем то, где не нужно.

04

Словарь предметов

Имена trait'ов становятся словарём: «это handheld tool», «это контейнер». Понятно из списка.

Два механизма, видны оба

Фрагменты (UItemFragment) — строительные блоки: Equippable, PickupActor, GrantsAbilities, Weapon, Container и т.д. Traits — рецепты, которые предзаполняют DA правильным набором этих фрагментов и определяют, что синтезируется автоматически. Дизайнер выбирает trait, видит внутри него набор фрагментов с дефолтами, и редактирует их поля напрямую.

03

// Куда это растёт

Когда подъедут будущие capability — это будут новые traits. Они добавятся только в те предметы, где нужны, не трогая остальные. Пример комбинированного предмета будущей волны:

DA_Item_SpecialSample · Traits
[0] Trait_BackpackItem // как лежит и берётся
EquippedActor · MeshAttachSocket · AbilitySet
[1] Trait_Scannable // как виден сканером
Stencil · NoiseScale
[2] Trait_X // специальная классификация
Kind · Severity

Три trait'а — три аспекта. Каждый собирается из своих параметров. У обычного предмета никакого Stencil или Severity в детайл-панели не появится — потому что нет такого trait'а.

04

// Что с существующими предметами

TL;DR для дизайнеров

Ничего делать не надо. Открыл предмет в editor — он уже в новом виде. Сохранил — формат закрепился.

Все существующие Item Data Asset мигрируются автоматически на загрузке. Сейв-формат игры, BP-классы предметов, иконки и меши — без изменений.

05

// FAQ

А если нужно для одного предмета своё поведение, которого нет в trait'е?
У каждого фрагмента внутри trait'а есть свои поля — правишь их прямо в DA. Если нужного фрагмента вообще нет в trait'е (например, добавить ApplyEffectOnAdd к обычному предмету), есть два пути: добавить фрагмент в дефолт trait-класса (применимо, если вариация осмысленна и для других предметов), либо создать новый trait-наследник в Blueprint с расширенным набором фрагментов.
Сломаются ли сейвы?
Нет. Сейв ссылается на DA по пути, миграция идёт прозрачно.
Можно ли продолжать создавать предметы по-старому, пока миграция не закончена?
Да. Новые предметы тоже автоматически попадут в новую модель на следующей загрузке.
Почему не сделать просто разные подклассы Item Data Asset?
Способности ортогональны: один предмет может быть одновременно обычным, сканируемым, особым и квест-item'ом. Подклассы — это однонаследная иерархия, которая не выражает «и то, и это, и третье». Traits — это «и то, и это» через композицию.
Куда писать с вопросами / правками?
Через внутренний канал команды.
