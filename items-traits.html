// CURRENT MODEL · INVENTORY

Items.
Traits + Archetypes.

Item Data Asset теперь собирается из editor-only traits и архетипов, а в runtime живёт на сгенерированных MergedFragments и MergedRuntimeActions. Эта страница описывает актуальную модель, а не старый план миграции.

01

// Коротко

Traits больше не заменяют одну «свалку полей» другой вложенной свалкой. Текущая система разделяет authoring, inheritance и runtime output.

01

Archetype

Родительский UItemDataAsset задаёт общий состав предмета: роль, слоты, pickup, equip, weapon setup и другие shared defaults.

02

Archetype Traits

Локальные копии унаследованных trait-слотов. Их можно править, а reset-to-default сравнивает значение с trait-ом из архетипа.

03

Traits

Локальные добавки предмета: например Stackable, Container или capability, которого нет в архетипе.

04

Build Output

MergedFragments и MergedRuntimeActions пересобираются из effective traits и сериализуются для runtime/cook.

Главное правило

Runtime не читает Traits. Traits — это editor authoring layer. Игровой код читает только построенные фрагменты через FindFragment<T>(), gameplay tags через IGameplayTagAssetInterface и runtime actions, активированные подсистемой.

03

// Правила композиции

ROOT → LEAF

Сборка идёт по цепочке архетипов

Сначала строится root archetype, затем каждый child. Цикл в цепочке логируется и обрывает рекурсию.

UCLASS

Override только по exact class

Trait заменяет унаследованный trait только если совпадает точный UClass. Subclass — это другой слот.

BUCKETS

Inherited отдельно от local

ArchetypeTraitOverrides — для inherited slots. Traits — только для локальных additions.

FRAGMENTS

Один фрагмент на class

Дубли одного fragment class запрещены, кроме mergeable fragments. UItemFragment_Tags mergeable и собирает role/capability/ad-hoc tags.

Build path
01 · Collect

CollectEffectiveTraits

Берёт traits parent-first, применяет Archetype Traits поверх inherited slots и добавляет local Traits.

02 · Build

BuildItemDefinition

Каждый effective trait кладёт authored fragments в UItemDefinitionContext или синтезирует implicit fragments.

03 · Merge

Context output

Context дублирует trait-owned fragments в DA, мерджит mergeable classes и пишет MergedFragments.

04 · Runtime

Serialized data

Game code работает с MergedFragments; runtime actions активируются один раз на GameInstance.

Validation

Валидатор ловит duplicate local trait classes, stale overrides, trait class в неправильном bucket и роль. Ожидается ровно один role trait в effective set; отсутствие или несколько ролей сейчас идут info-level сообщением.

// Текущий каталог traits

Broad role traits удалены. Роль — отдельная identity, остальные свойства разложены на узкие reusable traits.

identity

Item Role

Единственный role trait. Хранит RoleTag из Item.Role.* и синтезирует tags fragment при build.

common data

Display Info

DisplayName, Description, Icon. Заменяет старые root-поля DA.

common data

Tags

Ad-hoc tags дизайнера. Role/capability tags не кладём сюда вручную: их добавляют соответствующие traits.

common data

Pickup Actor

Pickup class и mesh overrides для spawn/pickup pipeline.

inventory

Slot Policy

Allowed inventory slots и bCanStoreInInventory. Заменяет старые ItemType и store flag.

equipment

Equippable · Equip Attach

Данные для предмета в руках и attach sockets. Разделены, чтобы weapon и обычные tools могли переиспользовать attach.

equipment

Grants Abilities

Ability set, который предмет добавляет владельцу при equip/use path.

placement

Placement Points

Luggage/container/cargo placement tags. Заменяет старые placement root-поля.

capability

Stackable · Container

Локальные additions для stack count и contained items. Также синтезируют capability tags.

runtime processor

Add Currency · Apply Effect

Владеют своим fragment и runtime action, который добавляет нужный actor component для обработки item-added events.

weapon

Weapon Setup

Не роль. Только weapon-specific fragments: arms animation, presentation, holder tags, firearm/ballistics/recoil/procedural fire или melee attack.

05

// Runtime model

UItemFragment остаётся data-only. Он может описать способность предмета и создать per-instance state через CreateInstanceFragment, но gameplay side effects вынесены в runtime processors/components.

  • Item instance: UInventoryItemInstance::SetItemDefinition проходит по MergedFragments и создаёт instance fragments.
  • Tags: UItemDataAsset отдаёт gameplay tags из built UItemFragment_Tags.
  • Runtime actions: traits пишут FGameFeatureActionSet, build копирует его в MergedRuntimeActions.
  • Activation: UItemRuntimeFeatureSubsystem сканирует известные item DA через Asset Registry и активирует actions один раз.
  • Dedup: одинаковые UGameFeatureAction_AddComponents схлопываются по actor class, component class, client/server flags и addition flags.
Пример

UItemTrait_AddCurrencyOnAdd добавляет UItemFragment_AddCurrencyOnAdd и runtime action на UItemAddedCurrencyComponent. Inventory публикует item-added event; компонент читает fragment и применяет результат.

06

// Как правильно авторить

  • Новый тип предмета: сначала делайте или выбирайте archetype с общей ролью и общими traits.
  • Изменить inherited value: правьте trait в Archetype Traits, не добавляйте trait того же класса в Traits.
  • Добавить orthogonal capability: добавляйте narrow trait в Traits, например Stackable или Container.
  • Weapon data: используйте Weapon Setup поверх shared role/display/slot/equip traits из weapon archetype.
  • Не редактировать: MergedFragments и MergedRuntimeActions. Это build output, он пересоздаётся.
07

// Миграция старых ассетов

Для дизайнеров

Старые Item Data Asset мигрируются на загрузке. Открыл asset в editor — увидел уже новую композицию. После save старый формат закрепляется как новый.

Migration path выбирает один из legacy archetypes по старым полям: luggage item, guard tool, PDA или currency. Затем переносит старые root-поля в inherited overrides или local traits, legacy ItemAddedActions — в matching capability traits, MaxStackCount — в Stackable, ContainedItems — в Container. Legacy поля пока оставлены в классе только для десериализации старых uassets.

08

// FAQ

Почему теперь два списка: Archetype Traits и Traits?
Потому что они означают разные вещи. Archetype Traits — это overrides уже существующих parent slots, с native reset-to-default. Traits — новые локальные capabilities, которых нет в archetype chain.
Что будет, если добавить в Traits trait того же класса, что уже есть в archetype?
Нормализация переносит такой trait в ArchetypeTraitOverrides, если inherited slot существует. Валидатор также подсвечивает trait classes, которые лежат не в том bucket.
Можно ли иметь несколько traits одного класса?
Нет, composition работает по exact UClass. Если нужно несколько логических entries одного типа, array должен жить внутри одного fragment/trait.
Куда делись broad traits вроде BackpackItem?
Их заменили narrow reusable traits. Роль теперь одна — Item Role; display, slots, pickup, equip, abilities, placement и capabilities живут отдельно. Так child asset может переопределить только нужный аспект, а не весь большой role trait.
Сломаются ли сейвы?
Нет. Сейвы продолжают ссылаться на item DA по asset path. Внутреннее устройство DA меняется на editor/build уровне.
