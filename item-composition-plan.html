Modular Items

Эта страница объясняет модульные предметы без необходимости читать код. Главная идея простая: апгрейд, attachment или компонент - это обычный предмет, установленный в слот другого предмета. Дизайнер настраивает слоты, модули и награды в редакторе; runtime хранит результат как состояние конкретного экземпляра предмета.

TL;DR
  • Host item declares named slots: Sensor, Lens, Optic, Trigger.
  • Module item is a normal item that can be installed into one of those slots.
  • Designer sets slot rules on the host and tags on module items.
  • Upgrade reward creates or grants a module item and installs it into the selected slot.
  • The result is saved on the concrete item instance, not only as an external purchase flag.

Start Here

One sentence

Модульный предмет - это предмет, который может содержать другие предметы в заранее названных слотах.

Например, улучшенный сканер - это не "сканер плюс купленный флаг". Это конкретный сканер, в котором установлен конкретный sensor module. Оружие с прицелом работает так же: weapon item содержит optic module.

Host item

Что улучшаем

Сканер, оружие, броня, контейнер или другой предмет, у которого есть слоты для модулей.

Module item

Что устанавливаем

Модуль тоже является предметом: sensor, lens, optic, barrel, trigger, battery, upgrade chip.

01

Setting Up A Modular Item

Дизайнерская работа делится на две части: у host item объявить слоты, а у module item поставить tags, которые позволяют ему встать в нужный слот. Runtime logic в fragments не пишется.

Step 1

Choose the host item

Это предмет, который будет получать модули: scanner, weapon, armor, backpack, device. У него должен быть обычный item data asset.

Step 2

Add a slots fragment

Для сканера это может быть Scanner Module Slots. Для оружия - Weapon Module Slots. Fragment содержит именованные поля, которые дизайнер видит как нормальные настройки.

Step 3

Set slot rules

У каждого слота можно указать readable name и required item tags. Если слот требует Module.Sensor, то item без этого tag не установится.

Step 4

Create module items

Каждый модуль - отдельный item data asset. Он получает display, icon, gameplay data и tags, которые подходят нужному слоту.

Upgrade Reward

What an upgrade does

Upgrade может купить/получить progression marker, но gameplay change должен приходить от module item, установленного в host item. Поэтому reward authoring использует resource Install Item Module.

Target Slot выбирается picker-ом: сначала fragment со слотами, потом конкретное поле. Ничего не вводится руками строкой.

Host Query choices

How the reward finds the host item

  • Equipped Item - удобно для предмета в руках.
  • Required Tags - удобно для уникального owned item с понятным role tag.
  • Exact Definition - безопасно только если у игрока не может быть двух таких предметов.
  • Item Id - точечный выбор конкретного экземпляра, обычно для специальных случаев.

04

Authoring Rules

Do
Why
Make upgrades as module items
The upgraded state belongs to the concrete item instance, so it saves, moves and debugs naturally.
Choose slots from picker
Picker points to a real slot field. Raw strings can silently break after renames.
Use clear module tags
Slot validation is easy to read: Sensor slot requires Module.Sensor, optic slot requires Module.Optic.
Make host queries unambiguous
If an upgrade can match two owned items, content should narrow the query or change the reward design.
Keep fragments data-only
Fragments describe slots and constraints. Installation logic stays in the shared system.
Avoid

Do not model a scanner upgrade only as IsPurchased(Upgrade.ScannerRange2) inside gameplay code. The upgrade can still have a progression marker, but the item's actual capability should come from the installed module item.

05

FAQ For Designers

Question

Is a module different from an item?

No. A module is an item used in a module slot. It can have display data, tags, gameplay data and save state like other items.

Question

Does the module stay with the item?

Yes. If a weapon has an optic installed, that optic belongs to that weapon instance's state. Moving or saving the weapon keeps its installed modules.

Question

Can modules be removed or swapped?

The architecture supports install/remove/swap as item slot operations. Specific UI, economy and balance rules are separate feature decisions.

Question

What if the reward finds several matching items?

That should fail. A reward must target one clear host item, otherwise the player could get a module installed into the wrong thing.

Question

Why not keep this as traits only?

Traits describe item type and allowed behavior. Installed modules are runtime state of one concrete item instance.

Question

Are purchase tags still useful?

Yes, for progression history and unlock checks. They should not be the only source of item capability when a concrete module can represent that capability.
